리뷰

지표가 좋으면 UX도 좋은걸까?

정성적인 평가나 유저의 만족도 같은 눈에 보이지 않는 영역을 가시화하기 위한 시도

→ 문제 제보 채널에 들어오는 유저 의견들을 주별, 월별 등의 시간순으로 분류

어드민 제품은 제품 자체가 복잡하다 보니 기능 개선 및 추가에 따라 문제 제보가 일어난 것으로도 생각할 수 있음

부정 경험 제보가 많아지기 시작한 시점 이전 3개월은 기술적인 부분이나 휴먼 에러와 관련된 문제 제보가 상위권, 근데 그 이후 3개월은 사용성이랑 관련된 부분이 상위권

이후 실제 유저들의 의견을 받음

만족도 조사를 시작으로 문항 구체화

그 안에 있는 패턴 찾아내기

→ 유저 인터뷰 진행

  • 자신의 상황에 해당하지 않는다고 생각되는 주제거나
  • 푸킹한 문구, 예상하지 못한 결과가 클릭 시 노출되면 광고라고 생각
  • 유저가 경험해 보지 않아서 상상할 수 없는 내용일 경우에도 마찬가지

⇒ 유저가 기대하는 내용이 나올 수 있도록 정직하게 문구를 작성하는 것이 중요하다는 것을 깨달음

이를 해결하기 위해 사례를 모아서 가이드라인 작성

UX 라이팅 팀에 템플릿 제공

광고 문법처럼 쓰인 문장인지 확인하는 검사기 제공

이 만족도 평가를 한눈에 볼 수 있게 대시보드 기획(담당자들이 스스로 점검이 가능하도록)

문화에 대한 문제까지 UX로 해결할 수 있을까?

토스팀은 기본적으로 자율과 책임의 문화를 갖고 있어 직접 의사결정권을 갖고 있기 때문에 동료들의 피드백 문화가 중요

→ 2월에 피드백 문화에 문제 제기가 들어옴

팀원들 인터뷰 후 가설을 세우고 검증을 하는 식으로 진행

  • 피드백 양이 너무 적다

    팀원 온보딩 프로세스 후 피드백을 제대로 주고 받은 경험이 없음

  • 당황스럽거나 공감하기 힘든 피드백을 받았을 때, 요청하기가 좀 겁난다

양과 질 둘중에 하나만 먼저 챙긴다면? → 양

양부터 늘려야 질에 대한 문제를 수집 가능, 질은 차후에 해결 가능

팀에서 진행하는 피드백 세션 참여

책 및 논문을 통해 리서치

  • 사람들은 피드백이 필요한 게 아님

    환경이 중요한 게 아니라 관리자가 관심을 갖고 지켜보는 게 중요

⇒ 피드백 자체가 관심의 표현, 본질적 가치를 잃어선 안 됨

  1. 갑자기 요청하는 게 아니라 미리 요청하고 관찰할 시간을 주면 더 쉬울 것이다
  2. 어떤 부분에서 피드백 받고 싶은지 명확하게 알려주면 상대방도 쉽게 답변할 수 있을 것이다

근데 요청하는 걸 쓰는데 시간이 꽤 걸림, 답변도 텍스트로 길게 쓰는 게 팀원분들한테도 부담

요청 플로우 개선

기간 선택, 목표 작성 페이지에서 오래 고민을 함

→ 목표를 직접 쓰는 건 빼고 누구나 해봤을 만한 고민을 선택하는 것만 남김

→ 이탈률이 너무 높음

기간 선택을 빼버림

인트로 페이지도 빠짐

피드백이라는 워딩이 무거운 듯해서 고민 나누기라는 단어를 채택

이탈률이 줄어들었음

답변도 어떻게 쉽게 쓸 수 있을까?

SBI기법을 적용해서 템플릿을 주고 작성하게 함

상황-행동-임팩트 순으로 이야기하는 것

  1. 큰 문제를 마주했을 때 어디서부터 시작해야될지 모르겠다면 인풋을 많이 넣어줘야 함

    통찰이 뇌의 먹이다

    → 근데 통찰을 얻으려면 계속 인풋을 넣어줘야 함

    → 인풋을 얻기 위해서는 용기가 필요

  2. 문제를 푸는 진짜 목적을 잊지 말아야 함

    피드백의 본질은 성장을 위한 것

어려운 기능, 꼭 설명해야 할까?

복잡한 기능을 이해하기 위해 기능 주변에 설명을 덕지덕지 붙이거나 PO와 가이드 문서를 보충하거나, 문의 대응을 따로함

→ 제품이 어려워짐

사용자 입장에서는 하루라도 빨리 실험을 통해 가설을 검증하고 더 나은 서비스를 내보내고 싶은데 전부 이해하고 사용하기에는 시간도 부족하고 비효율적

→ 사용자들이 설명이 너무 많아 이해가 어렵고 기능을 찾기 어려워 전문가에게 그냥 분석을 맡김

모두가 쉽게 데이터를 얻을 수 있어야 더 빨리 실험하고, 더 많은 가설들을 검증하고, 사용자들도 더 빨리 좋은 서비스를 경험할 수 잇음

  1. 어려운 개념을 이해시켜야 할 때

    특정 개념을 설명하는 데에 연연하는 것이 아니라 사용자들이 익숙한 어떤 경험을 꺼내서 보여주면 되는 게 아닐까?

    버튼을 클릭만 하면 원하는 대로 지표 수립이 가능

    특정 기능을 몰라도 사용할 수 있다

    설명을 하려 하지 않고, 그들이 익숙해하는 경험을 그대로 보여주는 것

  2. 조건을 이해시켜야 할 때

    옵션이 많은 기능의 경우 이 옵션들을 하나하나 보고, 이해하고, 보내고자 하는 대상들을 섬세하게 조정해야 함

    근데 이런 기능들을 발견하기도 어렵고 이해하기도 어려움

    불편함을 정리해보니

    • 어떤 옵션들이 어디에 있는지 모르겠고
    • 어떤 것은 체크하면 제외되고 포함되니까 헷갈리고
    • 설명이 너무 많아서 안 읽게 됨

    → 여러 페이지에 흩어져 있던 옵션들을 포함하기와 제외하기 버튼에 모아두고 각 기능들을 클릭하면 상세 화면을 만나서 기능을 사용할 수 있게 됨

    → 부정 보이스가 많이 들어옴/ 정리를 해야 된다고 생각해서 기능을 모음으로써 사용자들이 혼란스러워하는 원인이 됐기 때문

    안 찾아도 되고, 설명을 읽지 않아도 이해할 수 있도록 하려면?

    • 사용자에게 기능을 찾게 해야 하나? 그냥 필요할 때 딱 나타나면 되지 않나?

    • 사용자가 적극적으로 설명을 읽지 않으면 기능을 사용할 수 없는 구조라면, 사용자한테 어떤 이해를 떠넘기고 있던 게 아닐까?

      → 읽으면서 이해하고 제외할지 포함할지에 그냥 답할 수 있다면?

    발송 대상을 선택하자마자 관련된 기능들이 바로 보이도록

    읽자마자 답하면 되는 구조

  3. 가설을 최소한으로라도 확인을 해야 된다는 점

  4. 내가 정의하는 문제가 혹시 표면적인 문제가 아닌지 계속 의심을 해야 된다는 점

  5. 문제에서 뭔가 발견하기 어렵다면 이미 만들어둔 결과에서 힌트를 얻어볼 수 있다는 점

디자이너 없이 사용성을 지킬 수 있을까?

레거시 어드민: 회사를 인수하면서 받았던 어드민(30년 정도 됨)

  1. 사용이 너무 어려움

  2. 제품을 고칠 수 없음

    백단의 구조가 굉장히 복잡해 화면에 버튼 하나만 추가하려고 해도 여기저기서 오류가 남

기존 제품을 버리고 레거시 어드민들을 전부 다 새로 만들어야 함

⇒ 효율적인 제품을 만들어보자

  • 효율적인 제품: 복잡한 업무를 조금 더 쉽게 할 수 있도록 만드는 것

근데 개발 단계에서 너무 시간이 많이 걸림

⇒ 레거시 청산이라는 미션을 달성하려면 하나의 문제를 잘 풀기보다는 같은 시간 내에 훨씬 더 많은 문제를 더 효과적으로 풀어낼 방법이 필요했음

⇒ 레거시 화면만 보고도 모두가 같은 결과물을 떠올릴 수 있게 해야 된다

TDS: 레거시 화면을 보이는 그대로 TDS 컴포넌트로 다시 그리는 것

→ 근데 컴포넌트만 갈아끼우는 방식으로 이사를 하면 레거시 어드민의 쓰기 어려운 툴이라는 문제는 해결이 되지 않는다

→ 제품의 사용성 자체를 시스템화하는 게 필요하다

디자인 패턴들을 프라이머 에셋으로 만들어두고, 이 에셋만 사용해서 레거시 화면을 디자인

→ 테스트를 하면서 부족한 부분도 보완

→ 디자인 시스템과 가이드를 작성

  • 테이블에 대한 가이드

  • 기능을 잘 예측하게 하는 것

    레거시 어드민 화면을 보면서 어떤 시능들이 있는지 먼저 살펴보고 몇 가지 패턴을 가이드로 만듦

    → 개발자들이 상황에 맞게 가이드를 선택해서 만들 수 있도록 함

  • 화면의 심미성

    화면 전체에 대한 세심한 고려 필요

    더 큰 범위의 패턴을 만들기로 함
